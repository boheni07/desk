## 용어 정의

> **중요:** 아래 용어를 문서 전체에서 일관되게 사용한다.

| 용어 | 정의 |
|------|------|
| **처리희망일** | 고객담당자가 티켓 등록 시 입력하는 희망 처리 날짜 (기본값: 등록일 +5근무일) |
| **완료예정일** | 지원담당자가 수동 접수 시 입력하는 처리 완료 목표 날짜 |
| **연기요청일** | 연기요청 승인 시 확정된 새 완료 날짜 — 승인 후 처리기한으로 적용됨 |
| **처리기한** | 티켓의 현재 유효한 기한 — 상태에 따라 아래 값으로 자동 변경됨 |
| **처리기한 변경 이력** | 처리희망일 → 완료예정일 → 연기요청일 순으로 전환 기록 |
| **자동접수** | 인력 개입 없이 시스템이 자동으로 접수 처리하는 것 |
| **Main 담당자** | 프로젝트의 주 지원담당자 |
| **근무일/근무시간** | 평일 09:00~18:00 기준 (설정 가능), 공휴일 제외 |

### 처리기한 변경 규칙

| 시점 | 처리기한 값 | 비고 |
|------|-----------|------|
| 최초 등록 | 처리희망일 | 고객담당자 입력값 |
| 접수 (수동) | 완료예정일 | 기본값=처리희망일 (수정 안 해도 됨), 수정 시 처리희망일 기준 ±5근무일 범위만 선택 가능 |
| 접수 (자동) | 처리희망일 → 완료예정일로 자동 설정 | 사용자 수정 불가, 시스템이 처리희망일을 완료예정일로 자동 저장 |
| 연기요청 승인 | 연기요청일 | 승인된 새 완료 날짜 |

> **지연 감지:** 현재시각 > 처리기한 이면 처리중(IN_PROGRESS) → 지연중(DELAYED) 자동 전환

### 접수일시 기록 규칙

| 접수 유형 | 접수일시 기록값 |
|---------|--------------|
| **자동접수** (4근무시간 경과) | 티켓 등록일시 + 4근무시간 (계산값) |
| **자동접수** (지원담당 직접 등록) | 티켓 등록일시 |
| **수동접수** (지원담당 접수 버튼 클릭) | 지원담당자가 완료예정일 입력 후 확인한 실제 일시 |

| 장기체류 에스컬레이션 | **DELAYED 3근무일 이상** | Main 담당자 + 관리자에게 에스컬레이션 알림 |
| 처리희망일 기본값 | **등록일 +5근무일** | 달력 선택 시 기본값 |
| 완료예정일 범위 | **처리희망일 ±5근무일** | 수동접수 시 완료예정일 선택 범위 |
| 댓글 수정 제한 | **작성 후 10분 이내** | 본인 작성 댓글/메모 수정 가능 시간 |
| Push 구독 만료 | **90일 미사용** | 90일 이상 미사용 구독 배치 정리 |

## 1. 개요

### 1.1 배경 및 목적

nu-ServiceDesk는 고객사별 프로젝트를 기반으로 IT 서비스 요청을 티켓화하여 관리하는 플랫폼이다.
**한국 기업의 고객사-프로젝트-담당자 관계 구조**와 **계층형 승인 워크플로우**를 핵심으로 설계한다.

### 1.2 핵심 가치

- **Trust Building**: 티켓 처리 현황의 투명한 공유로 고객사와 신뢰 구축
- **프로젝트 중심**: 모든 티켓은 프로젝트에 귀속되어 고객별 히스토리 추적 가능
- **자동화**: 자동접수, 처리기한 기반 지연 감지, 연기 자동 승인

## 2. 사용자 역할

### 2.1 역할 정의

| 역할 | 구분값 | 설명 | 주요 기능 |
|------|--------|------|---------|
| **관리자** | `admin` | 시스템 전체 관리 | 모든 데이터 CRUD, 모든 티켓 수정, 전체 현황 관리 |
| **지원담당자** | `support` | 서비스 지원 수행 | 티켓 접수/처리/완료요청, 내부 메모 작성 |
| **고객담당자** | `customer` | 고객사 측 담당자 | 티켓 등록, 연기/완료 승인, 만족도 평가 |

### 2.2 역할별 접근 범위

| 역할 | 프로젝트 접근 | 티켓 수정 | 내부 메모 열람 |
|------|------------|---------|:----------:|
| 관리자 | 전체 | 전 상태에서 가능 | ✅ |
| 지원담당자 | 배정된 프로젝트만 | 접수 전만 가능 | ✅ |
| 고객담당자 | 배정된 프로젝트만 (복수) | 접수 전만 가능 | ❌ |

## 3. 핵심 도메인 구조

### 3.1 엔티티 관계

```
[사용자] (통합 테이블, 구분: admin / support / customer)
  - admin   : 고객사/부서 연결 없음
  - support : 고객사/부서 연결 없음
  - customer: 고객사(필수) + 부서(선택) 연결

[고객사]
└── 부서 (N)

[프로젝트]
├── 고객사 연계 (1, 필수)
├── 부서 연계 (0..1, 단일 부서)
├── 고객담당자 배정 (N) ← 반드시 해당 고객사 소속 customer 계정만
│   └── 고객담당자는 복수 프로젝트 배정 가능
├── 지원담당자 배정
│   ├── Main 담당자 (1, 필수)
│   └── 추가 지원담당자 (N, 선택)
└── 티켓 (N)

[시스템 설정]
├── 관리책임자 지정 (admin 계정, 1명 이상)
├── 근무시간 설정 (기본: 평일 09:00~18:00)
└── 알림 설정 (Phase 2)
```

### 3.2 티켓 등록 방식

| 등록 주체 | 등록 방법 | 프로젝트 선택 | 접수 처리 |
|----------|----------|-------------|---------|
| 고객담당자 | 직접 | 배정 프로젝트 자동 선택 (복수 시 목록 선택) | 지원담당 수동 접수 or 4근무시간 자동접수 |
| 지원담당자 | 전화/메일 대리 등록 | 배정 프로젝트 목록에서 선택 | 등록 즉시 자동 접수 |

## 4. 기능 요구사항

### 4.0 인증

#### FR-01 로그인 및 인증
- 로그인: ID/비밀번호 입력, 서버 세션 생성 (Redis 저장)
- 세션 유지: **Sliding Expiry 방식** — 마지막 API 요청 시각 기준 Redis TTL 8시간 자동 연장. 8시간 미활동 시 세션 만료 → 자동 로그아웃.
- 로그아웃: Redis 세션 삭제 (즉시 무효화)
- **세션 고정 공격 방어:** 로그인 성공 시 기존 세션 ID 폐기 → 새 세션 ID 발급 (session regeneration)
- 세션 쿠키: HttpOnly + Secure + SameSite=Strict, 세션 ID만 저장
- **비밀번호 변경:** 로그인 후 내 정보 페이지에서 변경 가능 (현재 비밀번호 확인 필수)
- **비밀번호 정책:** 최소 8자, 영문+숫자+특수문자 조합 필수
- **로그인 시도 제한:** 5회 연속 실패 시 15분 잠금 (brute force 방어)
- **비활성 계정 로그인 차단:** 비활성(inactive) 계정은 로그인 불가, 오류 메시지 표시
- 초기 비밀번호 강제 변경: 관리자가 생성한 계정은 최초 로그인 시 비밀번호 변경 요구
- **비밀번호 찾기:** Phase 1에서는 미제공 — 관리자 비밀번호 초기화(FR-03)로 대응. **Phase 2에서 셀프 비밀번호 재설정 기능 추가 예정** (등록 이메일로 재설정 링크 발송)

**로그인 이력 관리 (개인정보보호법 준수):**
- 모든 로그인 시도(성공/실패)를 `login_history` 테이블에 기록
- 기록 항목:

| 필드 | 설명 |
|------|------|
| 사용자 ID | 로그인 성공 시 연결된 사용자 (실패 시 null) |
| 입력된 로그인 ID | 사용자가 입력한 로그인 식별자 |
| 성공 여부 | 로그인 성공/실패 여부 (boolean) |
| IP 주소 | 요청 IP |
| User-Agent | 요청 브라우저/클라이언트 정보 |
| 실패 사유 | INVALID_PASSWORD / INACTIVE / LOCKED (성공 시 null) |
| 일시 | 로그인 시도 일시 |

- 개인정보보호법 준수를 위한 접근 이력 관리 목적

### 4.1 마스터 관리

#### FR-02 고객사 관리
- 고객사 CRUD (회사명, 사업자번호(선택), 주소, 연락처, 활성/비활성)
  - 사업자번호: 선택 입력, 입력 시 유니크 검증
- 고객사 하위 부서 CRUD (부서명, 부서코드)
- **비활성화 처리 (연쇄):**
  - 진행 중인 프로젝트/티켓이 있을 경우 알림창에 현황(프로젝트 N개, 진행중 티켓 N건) 표시 후 확인 요청
  - 확인 시: 소속 프로젝트 자동 비활성화 + 소속 customer 계정 자동 비활성화
  - 진행중 티켓(REGISTERED~COMPLETE_REQUESTED)은 목록에 표시 — 관리자가 개별적으로 취소/완료 수동 처리 (자동 종결 없음)

#### FR-03 사용자 계정 관리

사용자는 단일 테이블로 관리하며 `구분(type)` 필드로 역할을 구분한다.

**공통 필드:**

| 필드 | 설명 | 필수 |
|------|------|:----:|
| 로그인 ID | 로그인 식별자 (유니크) | ✅ |
| 비밀번호 | bcrypt 암호화 저장 | ✅ |
| 이름 | 실명 | ✅ |
| 이메일 | 알림 수신 이메일 (시스템 전체 유니크, 선택) | 선택 |
| 연락처 | 전화번호 (선택) | 선택 |
| 구분 | admin / support / customer | ✅ |
| 활성여부 | 활성 / 비활성 | ✅ |

> **[V1.1 변경]** 이메일, 연락처 필드가 선택(optional)으로 변경됨. 마이그레이션 `20260409081801_make_user_email_phone_optional` 적용 완료.

**구분 = customer 추가 필드:**

| 필드 | 설명 | 필수 |
|------|------|:----:|
| 고객사 | 소속 고객사 선택 | ✅ |
| 부서 | 소속 부서 (고객사 선택 후 목록 로드) | 선택 |

**기능 요구사항:**
- 관리자(admin) 계정만 사용자 CRUD 가능
- 구분 = customer 선택 시 고객사/부서 필드 동적 표시
- 비밀번호 초기화 (관리자 기능): 초기화 시 임시 비밀번호 생성 → 사용자 등록 이메일로 발송 → 첫 로그인 시 강제 변경
- **비활성화 처리:**
  - 처리 중인 티켓이 있을 경우 알림창에 현황 표시 후 종결 처리 안내
  - **지원담당자(support) 비활성화 시:** 해당 사용자가 Main 담당인 프로젝트 목록 표시 → Main 담당 변경 필수 (변경 전 비활성화 불가)
  - **고객담당자(customer) 비활성화 시:** 배정된 프로젝트에서 자동 해제. 프로젝트에 고객담당자 0명이 되면 관리자에게 경고 알림

#### FR-04 근무일 캘린더 (공휴일 관리)
- 관리자가 공휴일 수동 등록/삭제 (날짜 + 공휴일명)
- 연도별 조회/등록 지원
- 근무시간 기준: 평일 09:00~18:00 (설정 가능)
- 공휴일은 근무시간 계산 시 제외 (자동접수 4근무시간, 연기요청 8근무시간 등 전 영역 적용)

#### FR-05 카테고리 관리
- 관리자가 시스템 공통 카테고리 CRUD (카테고리명, 활성/비활성)
- **정렬순서(sortOrder):** Int 타입, default 0 — 관리자가 카테고리 정렬 순서를 설정 가능 (낮은 값이 앞에 표시)
- 모든 프로젝트/티켓에 동일 카테고리 목록 적용 (sortOrder 기준 정렬)
- 비활성화 시 신규 티켓 등록 시 선택 불가 (기존 티켓 유지)

> **[V1.1 추가]** 카테고리에 `sortOrder` 필드 추가. 관리자가 카테고리 노출 순서를 직접 제어 가능.

### 4.2 프로젝트 관리

#### FR-06 프로젝트 기본 정보
- 프로젝트 CRUD
- 필드: 프로젝트명, 프로젝트코드, 설명, 시작일, 종료일, 상태(활성/비활성)

- 지원담당자 배정: Main 담당자(1명, 필수) + 추가 지원담당자(N명, 선택)

#### FR-08 프로젝트 접근 제어

- **프로젝트 종료일 이후 신규 티켓 등록 불가** (기존 진행 중인 티켓 처리는 가능)
- **비활성화 처리:** 진행 중인 티켓이 있을 경우 알림창에 현황 표시 후 종결 처리 안내

### 4.3 티켓 관리

#### FR-09 티켓 번호 체계
- 자동채번 형식: `TK-YYYY-00000` (예: TK-2026-00001)
- 연도별 순번 증가 (연도 변경 시 00001부터 재시작)
- **채번 구현:** 별도 카운터 테이블 (`ticket_sequences`: year, last_number) + PostgreSQL `SELECT FOR UPDATE` 방식으로 동시 등록 시 유니크 보장
- **중복/갭 방지:** 트랜잭션 내에서 카운터 증가 → 티켓 INSERT를 원자적으로 실행. 실패 시 카운터 롤백.

#### FR-10 티켓 등록

**공통 입력 항목:**

| 필드 | 설명 | 필수 |
|------|------|:----:|
| 티켓번호 | TK-YYYY-00000 자동채번 | 자동 |
| 프로젝트 | 귀속 프로젝트 | ✅ |
| 제목 | 티켓 제목 | ✅ |
| 내용 | 문의/요청 내용 | ✅ |
| 카테고리 | 유형 분류 | ✅ |
| 우선순위 | 긴급/높음/보통/낮음 | ✅ |
| 처리희망일 | 기본값: 등록일 +5근무일, 수정 가능 | ✅ |
| 긴급처리 사유 | 우선순위=긴급 선택 시, 또는 처리희망일 < 등록일 +5근무일 시 표시 | 조건부 |
| 첨부파일 | 다중 파일 첨부 (파일 1개 최대 10MB, 티켓 전체 최대 50MB) | 선택 |
| 등록방법 | 직접 / 전화 / 메일 | 자동/선택 |

**처리희망일 입력 규칙:**
- 기본값: 등록일 +5근무일 (주말/공휴일 제외)
- 달력 직접 선택 가능
- **최솟값:** 등록일 +1근무일 이상 (과거 날짜 선택 불가)
- **최댓값:** 프로젝트 종료일 이내 (종료일 없는 프로젝트는 제한 없음)
- 선택 날짜 < 등록일 +5근무일 → **긴급처리 사유** 입력 필드 활성화 (필수)
- 우선순위 = **긴급** 선택 시에도 긴급처리 사유 입력 필드 활성화 (필수)

**첨부파일 제한:**
- 파일 1개 최대 10MB
- 티켓 전체 최대 50MB
- 허용 형식: 이미지(jpg/jpeg/png/gif/webp), 문서(pdf/doc/docx/xls/xlsx/ppt/pptx/txt)
- 실행 파일(exe, sh, bat 등) 및 압축파일 업로드 불가
- **저장소:** Cloudflare R2 (Presigned URL 방식 — 클라이언트가 서버에서 URL 발급 후 직접 R2에 업로드)

**고객담당자 등록:**
- 배정된 프로젝트 자동 선택 (복수 배정 시 목록에서 선택)
- 등록방법: "직접" 자동 기록

**지원담당자 대리 등록:**
- 배정된 프로젝트 목록에서 선택
- 고객담당자 선택 필수 (해당 프로젝트 배정된 담당자 중 선택)
  - **예외:** 해당 프로젝트에 고객담당자가 1명도 배정되지 않은 경우 대리 등록 불가 — 안내 메시지 표시: "프로젝트에 고객담당자를 먼저 배정해 주세요"
- 등록방법: "전화" 또는 "메일" 선택
- 등록 즉시 자동 접수 처리

#### FR-11 티켓 수정 제한
- **접수 전(등록 상태):** 등록자(고객담당자 또는 지원담당자)가 수정 가능
- **접수 후:** 수정 불가 — 변경사항은 공개 댓글로 소통
- **관리자:** 모든 상태에서 수정 가능 (수정 이력 자동 기록)
  - 수정 가능 필드: 제목, 내용, 카테고리, 우선순위, 담당자 배정
  - **상태 직접 변경 가능** (워크플로우 우회, 수정 사유 필수 입력)
  - **처리기한 직접 변경 가능** (변경 사유 필수 입력, 처리기한 변경 이력에 기록)
  - 모든 관리자 수정은 FR-20 통합 타임라인에 관리자 수정 이력으로 기록
- **취소:** 등록(REGISTERED) 상태에서 등록자(고객담당자 또는 지원담당자)만 취소 가능. 관리자도 취소 가능.
  - 취소 티켓은 일반 목록에서 노출되지 않으며 이력은 DB에 보존 (소프트 딜리트 방식)
  - 관리자 전용 취소 티켓 목록(쓰레기통) 별도 제공

#### FR-12 티켓 담당자 배정
- 프로젝트에 배정된 지원담당자 누구나 수동접수 가능
- **동시 접수 방지:** 두 지원담당자가 동시에 같은 티켓을 접수하는 경우 선착순 처리 (낙관적 락 — 먼저 접수 완료한 건만 반영, 후순위는 "이미 접수된 티켓입니다" 오류 반환)
- 수동접수 시 및 **IN_PROGRESS 중에도** 추가 지원담당자 배정 가능
- 배정된 추가 지원담당자는 **해제 가능** (관리자 또는 Main 담당자)
- 배정/해제 내역은 FR-20 통합 타임라인에 기록

#### FR-13 티켓 상태 목록

| 상태코드 | 상태명 | 설명 |
|---------|-------|------|
| `REGISTERED` | 등록 | 티켓 최초 생성 |
| `RECEIVED` | 접수대기 | 자동접수 후 지원담당자 확인 대기 (수동접수는 해당 없음) |
| `IN_PROGRESS` | 처리중 | 처리기한 설정 후 진행 중 |
| `DELAYED` | 지연중 | 처리기한 초과 자동 전환 (배치 감지 또는 요청 반려 후 복귀 시) |
| `EXTEND_REQUESTED` | 연기요청 | 연기 승인 대기 중 |
| `COMPLETE_REQUESTED` | 완료요청 | 완료 승인 대기 중 |
| `SATISFACTION_PENDING` | 평가대기 | 완료요청 승인 후 만족도 평가 대기 중 |
| `CLOSED` | 종료 | 만족도 평가 완료 후 최종 종료 |
| `CANCELLED` | 취소 | 등록자가 접수 전 취소 처리 — 일반 목록에서 숨김, 관리자만 조회 가능 |

#### FR-14 티켓 상태 전이 규칙

```
[등록 REGISTERED]
  │
  ├── [수동접수] 지원담당자가 '접수' 버튼 클릭
  │     → 완료예정일 입력 모달 팝업
  │       기본값=처리희망일 (수정 안 해도 됨)
  │       수정 시 → 처리희망일 기준 ±5근무일 범위 내에서만 날짜 선택 가능
  │     → 확인 클릭 시 IN_PROGRESS 직행 (RECEIVED 단계 없음)
  │       처리기한 = 완료예정일, 접수일시 = 확인한 실제 일시
  │
  └── [자동접수 RECEIVED] — 두 가지 경로
        ├── 자동①: 지원담당자 직접 등록 → RECEIVED → IN_PROGRESS 즉시
        │          접수일시 = 등록일시
        └── 자동②: 근무시간 누적 4시간 경과 → RECEIVED
                   완료예정일 = 처리희망일 자동 설정 (수정 불가)
                   처리기한 = 완료예정일(=처리희망일)
                   접수일시 = 등록일시 + 4근무시간 (계산값)
                   → 해당 프로젝트에 배정된 지원담당자 또는 관리자가 티켓 상세 페이지 열람 시 자동 전환:
                      · 처리기한 미초과 → IN_PROGRESS 자동 전환
                      · 처리기한 초과 → DELAYED 자동 전환
                      (상세 페이지 진입 시 클라이언트가 RECEIVED 상태 감지 → 별도 POST `/api/tickets/{id}/confirm` 호출하여 상태 전환. GET 조회와 상태 변경을 분리하여 REST 원칙 준수. **멱등성 보장:** 조건부 UPDATE `WHERE status = 'RECEIVED'` + affected rows 체크로 중복 호출 무시. 복수 담당자 동시 열람 시에도 최초 1회만 전환.)
  ↓
[처리중 IN_PROGRESS]
  ↓ 처리기한 초과 시 배치 자동 전환
[지연중 DELAYED]

[처리중 IN_PROGRESS] 가능 액션:
  ├── 연기요청 (조건: 처리기한 8근무시간 전까지) → EXTEND_REQUESTED
  └── 완료요청 → COMPLETE_REQUESTED

[지연중 DELAYED] 가능 액션:
  └── 완료요청만 가능 → COMPLETE_REQUESTED
  (※ 의도적 설계: DELAYED → IN_PROGRESS 복귀 경로 없음. 지연은 패널티 성격이므로 연기요청/처리재개 불가, 완료요청으로만 진행)

[연기요청 EXTEND_REQUESTED]
  ├── 승인 → IN_PROGRESS (처리기한 = 연기요청일)
  └── 반려 → IN_PROGRESS (처리기한 초과 시 즉시 DELAYED)

[완료요청 COMPLETE_REQUESTED]
  ├── 승인 → SATISFACTION_PENDING
  ├── 반려(1~2회) → 이전 상태(IN_PROGRESS or DELAYED) 복귀
  └── 3회차 → 자동승인 → SATISFACTION_PENDING

[평가대기 SATISFACTION_PENDING]
  ├── 만족도 평가 완료 → CLOSED
  └── 5근무일 미평가 → CLOSED (자동)

[등록 REGISTERED] 취소 가능 액션:
  └── 취소 → CANCELLED (일반 목록 숨김)

> **RECEIVED 취소 불허:** RECEIVED 상태에서는 취소 불가. 자동접수 후에는 완료요청 워크플로우를 통해서만 종결 가능.

> **장기 체류 방지:** RECEIVED 또는 DELAYED 상태에서 **3근무일 이상** 담당자 미확인(confirm 미호출 또는 액션 없음) 시, Main 담당자 및 관리자에게 Web Push 에스컬레이션 알림 발송. (배치 주기: 매일 09:00)
```

#### FR-15 자동접수 스케줄러

- 근무시간 기준: 평일 09:00~18:00 (설정 가능, 공휴일 제외)
- **배치 주기:** 1분 단위 실행 (지연감지 FR-16과 동일 주기, 자동접수 → 지연감지 순서로 순차 실행하여 경합 방지)
- 조건: 등록(REGISTERED) 상태 + 근무시간 누적 **4시간 경과**
- **카운트 시작 시점:** 등록일시가 근무시간 내이면 즉시 시작, 근무시간 외(야간/주말/공휴일)이면 **다음 근무일 09:00부터** 카운트 시작
- **완료예정일 자동 설정:** 처리희망일을 완료예정일로 자동 저장 (사용자 수정 불가)
- **처리기한 업데이트:** 처리기한 = 완료예정일(=처리희망일)
- **접수일시 기록:** 티켓 등록일시 + 4근무시간 (실제 처리 시점이 아닌 계산값)
- 자동접수 완료 시 RECEIVED 상태로 전환 (완료예정일/처리기한 자동 설정)
- 이후 지원담당자가 해당 티켓 상세 확인 시 IN_PROGRESS 자동 전환
- 지원담당자 직접 등록의 경우: 등록 즉시 RECEIVED → IN_PROGRESS 전환 (등록 행위가 확인을 포함)

#### FR-16 지연 자동 감지

- 배치 주기: 1분 단위 (자동접수 FR-15 배치 실행 완료 후 순차 실행 — 동일 트랜잭션 내에서 경합 방지)
- **감지 대상 상태:** 처리중(IN_PROGRESS) 및 접수대기(RECEIVED)
- 조건: 현재시각 > 처리기한
- 자동 전환:
  - IN_PROGRESS → DELAYED
  - **RECEIVED → DELAYED** (자동접수 후 지원담당자가 장기간 확인하지 않아 처리기한 초과 시)
    - 이후 담당자가 DELAYED 상태 티켓을 확인할 경우 별도 처리 필요 (상태 전환 없음, 댓글/완료요청만 가능)
- **대기 중 상태 처리:** EXTEND_REQUESTED, COMPLETE_REQUESTED는 배치 감지 대상 아님
  - 단, 반려 후 IN_PROGRESS 복귀 시점에 현재시각 > 처리기한이면 **즉시 DELAYED 전환** (FR-17, FR-18 참조)
- **DELAYED 전환 트리거 두 가지:**
  1. 배치 감지 (1분 주기, IN_PROGRESS 또는 RECEIVED 상태에서)
  2. 요청 반려 후 IN_PROGRESS 복귀 즉시 실시간 재확인

#### FR-17 연기요청

**신청 조건:**
- 상태: 처리중(IN_PROGRESS)
- 시점: 처리기한 기준 **8근무시간(1근무일) 전까지** 신청 (잔여 근무시간 ≥ 8일 때 가능)
- 횟수: 티켓당 **최대 1회**
- **완료요청 반려 후에도 가능:** 완료요청이 반려되어 IN_PROGRESS로 복귀한 상태에서, 이전에 연기요청을 사용하지 않았으면 연기요청 가능

**요청 내용:** 새 완료날짜(연기요청일), 연기 사유

**연기요청일 선택 범위:**
- 현재 처리기한 이후 날짜만 선택 가능 (이전 날짜 선택 불가)
- 현재 처리기한 기준 **최대 10근무일(2주) 이내**

**자동 승인:**
- 신청 후 **4근무시간 내** 고객담당자가 반려하지 않으면 **자동 승인**
- **배치 주기:** 1분 단위 실행 (자동접수/지연감지와 별도 스케줄, 근무시간 경과 판정)
- 자동 승인 시 처리기한 = 연기요청일로 업데이트 → IN_PROGRESS 유지
- **자동 승인 시 고객담당자에게 Web Push 알림 발송** (FR-23 참조)

**수동 처리:**
- 승인: 처리기한 = 연기요청일(미래 날짜) → IN_PROGRESS 유지
- 반려: 반려 사유 필수 입력 → IN_PROGRESS 복귀 + 사유 이력 기록
  - **반려 복귀 후 현재시각 > 처리기한이면 즉시 DELAYED 자동 전환**

#### FR-18 완료요청 (에스컬레이션 규칙)

완료요청은 최대 3회 관리되며 반려 횟수에 따라 에스컬레이션된다.

| 요청 회차 | 지원담당자 | 고객담당자 반려 시 | 자동 처리 |
|----------|----------|----------------|---------|
| **1회차** | 처리 결과 내용 입력 | 반려 사유 필수 입력 | — |
| **2회차** | 완료요청 내용 **필수** 재입력 | 반려 사유 필수 입력 + **관리책임자 Web Push 알림 발송** | — |
| **3회차** | 내용 입력 후 요청 | — | **자동 승인** → 만족도 평가 → 종료 |

- 관리책임자 알림: **시스템 설정 > 관리책임자** 메뉴에서 등록된 admin 계정(1명 이상 필수 설정)으로 Web Push 발송. 미설정 시 관리자 대시보드에 경고 배너 표시. **Fallback:** 관리책임자 미설정 시 전체 admin 계정에게 알림 발송.
- **3회차 자동승인 시 고객담당자에게 Web Push 알림 발송** (FR-23 참조)
- 자동 승인(3회차) 이력에 "3회차 자동승인" 기록
- **회차 카운터 저장:** 티켓 테이블에 `complete_request_count` 필드로 관리 (1~3). 완료요청 제출 시 +1, 에스컬레이션 판단에 사용.
- **회차 카운터 리셋:** 연기요청 승인 후 새로운 완료요청은 1회차부터 재시작
- **반려 후 상태 복귀:** IN_PROGRESS 또는 DELAYED로 복귀
  - 복귀 시 현재시각 > 처리기한이면 즉시 DELAYED 자동 전환

**고객담당자 부재 시 관리자 대리 승인:**
- 프로젝트에 활성 고객담당자가 0명인 경우, 관리자(admin)가 연기/완료 승인/반려 권한을 대리 행사
- 관리자 대시보드에 "고객담당자 부재 — 대리 승인 필요" 알림 표시
- 대리 승인 이력에 "관리자 대리승인" 기록

#### FR-19 만족도 평가

- 트리거: 완료요청 승인 직후 SATISFACTION_PENDING 상태로 전환
- 평가 항목: 별점(5점), 코멘트(선택)
- 평가 완료 → CLOSED
- **미평가 자동 종료:** SATISFACTION_PENDING 전환 후 **근무일 5일** 경과 시 자동 CLOSED 처리

- 알림 권한 미허용 사용자: DB 저장 + 알림 센터(FR-24)에서만 확인 가능

**Phase 2 추가:** 이메일 발송 (Web Push와 병행, 사용자 설정으로 수신 여부 선택 가능)

#### FR-24 알림 센터 (Notification Center)

모든 대시보드(FR-25, FR-26, FR-27) 공통 헤더에 알림 센터를 제공한다.

**헤더 벨 아이콘:**
- 읽지 않은 알림 수 뱃지 표시 (최대 99+)
- 뱃지 수 갱신: **30초 주기 폴링** (`/api/notifications/unread-count`)
- 클릭 시 알림 드롭다운 패널 열림

**알림 드롭다운 패널:**
- 최근 알림 20건 표시 (최신순)
- 읽지 않은 알림: 강조 표시 (배경색 구분)
- 알림 항목 클릭 → 해당 티켓 상세 페이지로 이동 + 읽음 처리
- "전체 읽음" 버튼
- "알림 전체 보기" 링크 → 전체 알림 목록 페이지 (`/notifications`)

**알림 전체 목록 페이지 (`/notifications`):**
- 읽음/읽지않음 필터
- 알림 유형별 필터
- 무한스크롤 또는 페이지네이션
- 알림 개별 삭제 (소프트 딜리트)
