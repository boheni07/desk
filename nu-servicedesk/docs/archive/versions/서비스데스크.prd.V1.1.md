# nu-ServiceDesk PRD (Product Requirements Document)

> **"Trust Building Platform"** -- 한국 시장 특화 티켓 기반 서비스 지원 플랫폼

| 항목 | 내용 |
|------|------|
| 문서 버전 | V1.1 |
| 작성일 | 2026-04-09 |
| 기능명 | nu-ServiceDesk |
| 상태 | In Development (Phase 1 구현 완료) |

## Executive Summary

| 관점 | 설명 |
|------|------|
| **Problem** | 한국 기업은 글로벌 서비스데스크 솔루션(Zendesk, Freshdesk)의 불완전한 한국어 지원, 카카오톡/SMS 알림 부재, 한국형 계층적 승인 구조 미지원, 개인정보보호법 비준수 등으로 인해 서비스 운영에 마찰을 겪고 있다. |
| **Solution** | 완전한 한국어 UI, 카카오톡 알림, 계층형 승인/보고 워크플로우, 국내 데이터 레지던시를 기본 제공하는 SaaS 티켓 기반 서비스데스크를 구축한다. |
| **Functional UX Effect** | 서비스 요청자는 카카오톡으로 실시간 상태 알림을 받고, 담당자는 한국어 자동 분류와 SLA 기반 대시보드로 응답 시간을 50% 단축하며, 관리자는 계층형 승인 워크플로우로 보고 체계를 일원화한다. |
| **Core Value** | 글로벌 솔루션 대비 30-50% 낮은 비용으로 한국 규정 완전 준수와 한국 비즈니스 문화에 최적화된 서비스데스크 경험을 제공하여, 국내 중소/중견기업의 IT 서비스 관리 디지털 전환을 가속한다. |

## 2. PM Strategy: Value Proposition & Business Model

### 2.1 JTBD 6-Part Value Proposition

#### Jobs-To-Be-Done Statement

> **[상황]** IT 지원 요청이 이메일, 전화, 메신저로 분산되어 관리되지 않을 때,
> **[동기]** 모든 서비스 요청을 하나의 시스템에서 추적하고 SLA를 준수하고 싶다.
> **[기대 결과]** 서비스 품질이 투명하게 측정되고, 요청자 만족도가 향상되며, 팀의 업무 부하가 균등하게 분배된다.

#### 6-Part VP Canvas

| Part | 내용 |
|------|------|
| **1. Target Customer** | 한국 중소/중견 기업(50-500명)의 IT 운영팀, 고객지원팀, 총무/시설관리팀 |
| **2. Problem** | 글로벌 서비스데스크는 한국어 지원 불완전, 카카오톡 미연동, 한국형 승인 구조 미지원, 비싼 가격, 해외 서버 데이터 주권 문제가 있다 |
| **3. Promise** | 한국 기업에 최적화된, 합리적 가격의, 규정을 준수하는 서비스데스크를 제공한다 |
| **4. Proof** | Beta 고객 SLA 준수율 95%+, 카카오톡 알림 응답률 이메일 대비 2x, PIPC 준수 인증 |
| **5. Power (Moat)** | 한국어 네이티브 UX + 카카오톡 생태계 연동 + 국내 데이터 레지던시 + 한국형 워크플로우 = 글로벌 솔루션이 쉽게 복제할 수 없는 조합 |
| **6. Profit Model** | SaaS 구독 (agent당 월과금) + 프리미엄 티어 (자동화/AI) + Enterprise (온프레미스/전용 환경) |

- 전략적 시사점: 비IT 부서까지 포용하는 범용성으로 차별화

**4. ManageEngine ServiceDesk Plus**
- 강점: ITIL 기반 완성도, 자산관리 통합, 온프레미스 옵션
- 약점: UI 구식, SaaS 전환 느림, 한국어 지원 제한
- 전략적 시사점: 모던 UX + SaaS 네이티브로 대체 타겟

**5. Zoho Desk**
- 강점: Zoho 생태계, 합리적 가격, 커스터마이징 유연
- 약점: 한국 시장 인지도 낮음, 한국어 제한적, 카카오톡 미지원

### 5.3 User Stories (Tier 1 MVP)

#### Epic 1: 티켓 관리

```
US-1.1: As a 서비스 요청자,
        I want to 웹 포털에서 티켓을 생성할 수 있도록,
        So that 내 서비스 요청이 체계적으로 접수되고 추적된다.
        Acceptance Criteria:
        - 제목, 설명, 카테고리, 우선순위, 첨부파일 입력 가능
        - 생성 즉시 티켓 번호 부여 및 확인 이메일 발송
        - 마이 티켓 목록에서 상태 확인 가능

US-1.2: As a 에이전트,
        I want to 할당된 티켓 목록을 우선순위와 SLA 잔여시간 기준으로 정렬하여 볼 수 있도록,
        So that 긴급한 티켓을 먼저 처리할 수 있다.
        Acceptance Criteria:
        - SLA 임박 티켓 상단 고정
        - 상태/우선순위/카테고리 필터링
        - 벌크 할당/상태 변경

US-1.3: As a 에이전트,
        I want to 티켓에 내부 메모와 공개 응답을 구분하여 작성할 수 있도록,
        So that 팀 내부 논의와 고객 커뮤니케이션이 분리된다.
        Acceptance Criteria:
        - 내부 메모: 에이전트/관리자만 조회
        - 공개 응답: 요청자에게 이메일/카카오톡 알림 발송
        - 타임라인에 구분 표시

US-1.4: As a 관리자,
        I want to 티켓 카테고리와 우선순위 체계를 설정할 수 있도록,
        So that 조직에 맞는 분류 체계를 운영한다.
        Acceptance Criteria:
        - 카테고리 CRUD (최대 3depth)
        - 우선순위별 SLA 자동 매핑
        - 기본 템플릿 제공 (IT지원, 시설관리, 인사)
```

#### Epic 2: SLA 관리

```
US-2.1: As a 관리자,
        I want to 우선순위별 SLA 정책을 정의할 수 있도록,
        So that 서비스 수준 기준이 명확히 운영된다.
        Acceptance Criteria:
        - 우선순위별 응답/해결 시간 설정
        - 영업일 기준 계산 (공휴일 캘린더)
        - 에스컬레이션 규칙 (1단계: 알림, 2단계: 재할당)

US-2.2: As a 에이전트,
        I want to SLA 잔여 시간을 실시간으로 확인할 수 있도록,
        So that SLA 위반을 사전에 방지한다.
        Acceptance Criteria:
        - 티켓 목록/상세에 SLA 타이머 표시
        - 80% 경과 시 주의 알림 (노란색)
        - 위반 시 긴급 알림 (빨간색) + 에스컬레이션
```

#### Epic 3: 사용자 및 접근 관리

```
US-3.1: As a 관리자,
        I want to 역할 기반으로 사용자 권한을 관리할 수 있도록,
        So that 적절한 접근 통제가 이루어진다.
        Acceptance Criteria:
        - 4개 기본 역할: Admin, Agent, Viewer, Requester
        - 역할별 메뉴/기능 접근 권한 매트릭스
        - 팀/부서 단위 할당 관리
```

### 10.1 Tech Stack

| Layer | Technology | 근거 |

| **Hosting** | Vercel (Frontend) + AWS Seoul (Backend/DB) | 글로벌 CDN + 국내 데이터 레지던시 |
| **Monitoring** | Sentry, Vercel Analytics | 에러 추적, 성능 모니터링 |

## 부록: Phase 1 구현 완료 현황 (2026-04-09 기준)

### A.1 구현 완료된 기능 목록

#### API Routes (전체 구현 완료)

| 도메인 | 엔드포인트 |
|--------|-----------|
| **인증** | login, logout, session, 비밀번호 변경 |
| **고객사** | CRUD + deactivate + 부서 CRUD |
| **사용자** | CRUD + 비밀번호 초기화 |
| **프로젝트** | CRUD + 멤버 관리 |
| **카테고리** | CRUD |
| **공휴일** | CRUD |
| **시스템 설정** | 관리책임자 CRUD |
| **티켓** | 전체 워크플로우 (등록/접수/처리/연기/완료/취소/만족도/댓글) |
| **첨부파일** | Presigned URL 발급 + 삭제 |
| **알림** | 조회/읽음/전체읽음/삭제 |
| **Push 구독** | 구독 등록/해제 관리 |
| **대시보드** | 통계 데이터 조회 |

#### BullMQ 배치 잡 (7개 확정)

| 잡 이름 | 실행 주기 | 역할 |
|---------|----------|------|
| `auto-receive` | 1분 | 티켓 자동 접수 처리 |
| `delay-detect` | 1분 | 지연 티켓 감지 및 상태 전환 |
| `extend-auto-approve` | 1분 | 연장 요청 자동 승인 |
| `satisfaction-close` | 1시간 | 만족도 미응답 티켓 자동 종결 |
| `project-deactivate-notify` | 1일 | 비활성 프로젝트 알림 발송 |
| `customer-zero-warning` | 1일 | 고객사 티켓 0건 경고 알림 |
| `stale-escalation` | 1일 | 장기 미처리 티켓 에스컬레이션 |

#### 핵심 라이브러리

| 라이브러리 | 설명 |
|-----------|------|
| **BusinessHoursEngine** | 근무시간 기반 SLA 계산 (date-fns, Asia/Seoul 타임존) |
| **TicketStateMachine** | 9개 상태, 17개 이벤트 기반 티켓 상태 관리 |
| **Redis 세션** | Sliding Expiry 방식, 만료 시간 8시간 |
| **Web Push (VAPID)** | 브라우저 푸시 알림 발송 |
| **Cloudflare R2** | Presigned URL 기반 첨부파일 관리 |
