## 1. OST 실현도 매트릭스

PRD Step 5(OST)에 정의된 7개 기회영역(O1~O7 — OST 트리 기준, Step 1 O1~O8과 매핑)과 각 Solution의 구현 상태를 분석한다.

> 참고: PRD Step 1(브레인스토밍)에는 O8(전환 마이그레이션)까지 8개 기회영역이 있으나, Step 5 OST 트리는 7개로 통합 정의됨. 이 보고서는 Step 5 OST를 기준으로 분석하되 Step 1 O8은 별도 항목으로 포함.

## 2. 가정 검증 상태 업데이트

PRD에 정의된 7개 핵심 가정(A1~A7)의 검증 가능성을 구현 결과 기준으로 평가한다.

---

### A1 — 한국 중소/중견 기업의 전환 의사 | 영향도 5 × 불확실성 3 = 우선순위 15

**PRD 상태**: 미검증  
**검증 방법**: Landing Page Test (100건/월 사전등록)  
**구현 기여도**: 높음

구현된 시스템은 A1 검증을 위한 제품 완성도를 충족함. 온보딩 마법사(F1.11) 4단계, 역할별 대시보드(F1.6), 한국어 완전 UI(F1.4)로 실제 사용성 증거를 제공 가능. Beta 50팀 온보딩 준비 완료 선언이 곧 A1 검증 트리거.

**현재 상태**: 기술적 준비 완료. A1 검증은 Beta 출시 후 실사용자 확보로만 가능.  
**권고**: Beta 출시 즉시 OnboardingState 완료율 추적 데이터를 A1 프록시 지표로 활용.

---

### A2 — 카카오톡 알림의 응답률 향상 효과 | 영향도 4 × 불확실성 2 = 우선순위 8

**PRD 상태**: 미검증 (Phase 2 A/B 테스트 계획)  
**구현 기여도**: 낮음 (카카오 미구현)

Web Push 21개 알림 타입이 구현되어 이메일/카카오톡과의 A/B 테스트 기준선(Control) 역할 가능. Phase 2 카카오 연동 후 알림 응답률 비교를 위한 NotificationType enum 구조가 이미 확장 가능하게 설계됨.

**현재 상태**: Web Push 응답률 측정 인프라 완비. 카카오 미구현으로 A2 직접 검증 불가.  
**권고**: Phase 1에서 Web Push 응답률(알림 수신 후 티켓 조회 시간)을 측정하여 Phase 2 A/B 테스트 베이스라인 확보.

---

### A3 — 국내 데이터 레지던시가 구매 결정의 핵심 요인 | 영향도 4 × 불확실성 3 = 우선순위 12

**PRD 상태**: 미검증 (고객 인터뷰 15명 계획)  
**구현 기여도**: 중간

개인정보보호법 준수 코드(로그인 이력 1년 보존, HTTPS, HSTS), AWS Seoul Region 기본 배포 아키텍처가 A3 주장의 "실물 증거"로 활용 가능. 영업/마케팅에서 "코드 레벨에서 규정이 내재화됨"을 증명 자료로 활용 가능.

**현재 상태**: 기술적 증거 확보. 실제 구매 결정 영향력은 고객 인터뷰로만 검증 가능.  
**권고**: Beta 고객사 온보딩 시 "데이터 위치가 도입 결정에 얼마나 영향을 주었나" JTBD 인터뷰 질문을 포함.

---

### A4 — 가격이 전환의 핵심 동인 | 영향도 5 × 불확실성 2 = 우선순위 10

**PRD 상태**: 미검증 (Van Westendorp 조사 계획)  
**구현 기여도**: 중간

Free Plan 제한 상수(`FREE_PLAN_MAX_USERS=3`, `FREE_PLAN_MAX_STORAGE_GB=5`)와 업그레이드 프롬프트 5조건이 구현됨. 사용자가 업그레이드 프롬프트에 어떻게 반응하는지(클릭률, 전환 여부)가 A4의 직접 프록시 지표.

**현재 상태**: 업그레이드 트리거 이벤트 로그 수집 가능. 실제 전환율은 Beta 운영 후 측정.  
**권고**: 업그레이드 프롬프트 5개 조건별 노출/전환 이벤트를 별도 테이블에 로깅하는 Analytics 후크 추가 고려.

---

### A5 — 한국어 AI 분류 80% 이상 정확도 | 영향도 3 × 불확실성 4 = 우선순위 12

**PRD 상태**: 미검증  
**구현 기여도**: 없음 (AI 미구현)

Phase 1에서 AI 분류 기능 자체가 미구현(F3.1, Phase 3). 단, 티켓 카테고리 시스템과 구조화된 티켓 데이터(제목, 설명, 카테고리, 우선순위)가 향후 AI 학습 데이터셋의 원형을 제공.

**현재 상태**: A5 검증 불가. 데이터 축적 단계.  
**권고**: Beta 단계에서 수동 분류 데이터를 충분히 축적(최소 1,000건)한 후 Phase 3 AI 분류 정확도 검증에 활용.

---

### A6 — 계층형 승인 워크플로우가 50% 이상 기업에서 필수 | 영향도 4 × 불확실성 3 = 우선순위 12

**PRD 상태**: 미검증 (프로토타입 테스트 계획)  
**구현 기여도**: 높음

연기요청(1회)·완료요청(3회 에스컬레이션)·대리 승인·자동 승인 워크플로우가 완전 구현됨(F1.8). Phase 1 자체가 A6의 "Prototype Test" 역할. Beta 고객사가 실제로 승인 워크플로우를 사용하는지 여부가 A6 검증의 핵심.

**현재 상태**: 실물 프로토타입으로 A6 검증 가능 상태. PRD 실험 계획(태스크 완료율 90%+)을 Beta 사용 데이터로 측정 가능.  
**권고**: 고객 온보딩 후 30일 내 "연기요청 또는 완료요청 사용 비율"을 A6 검증 지표로 설정.

### A7 — SaaS 전환 고려 기업의 국산 솔루션 선호도 | 영향도 3 × 불확실성 4 = 우선순위 12

**PRD 상태**: 미검증  
**구현 기여도**: 낮음 (브랜드/마케팅 영역)

구현된 제품 자체보다는 GTM 실행 결과로 검증 가능한 가정. Beachhead 세그먼트(IT/SaaS 스타트업) 확보 여부가 A7의 간접 지표.

**현재 상태**: A7 검증 불가 (운영 전).  
**권고**: Beta 신청자에게 "기존에 고려한 대안은?"을 물어 국산/해외 솔루션 경쟁 현황 데이터 수집.

## 3. ICE 우선순위 vs 구현 순서 정합성

### ICE 순위와 구현 Phase 대조

| ICE 순위 | 기회영역 | ICE Score | 구현 Phase | 정합성 |
|:--------:|---------|:---------:|:----------:|:------:|
| 1 | O1 한국어 완전 지원 | 504 | Phase 1 Core (F1.4) | 일치 |
| 2 | O5 비용 장벽 해소 | 486 | Phase 1 Core (F1.12) | 일치 |
| 3 | O2 카카오톡 알림 | 448 | **Phase 2** (F2.1) | 부분 불일치 |
| 4 | O3 계층형 승인/보고 | 336 | Phase 1 Core (F1.8) | 일치 |
| 5 | O4 데이터 주권 | 280 | Phase 1 Core (F1.3, OWASP) | 일치 |
| 6 | O8 전환 마이그레이션 | 210 | Phase 2~3 (F2.9, F3.8) | 일치 |
| 7 | O6 AI 한국어 분류 | 140 | Phase 3 (F3.1) | 일치 |
| 8 | O7 국내 생태계 연동 | 108 | 미정 (Phase 3+) | 일치 |

### 정합성 분석

**일치 항목 (7/8)**: ICE 점수와 구현 Phase가 높은 정합성을 보인다. 특히 ICE 1위(한국어 UI), ICE 4위(계층형 승인), ICE 5위(데이터 주권)가 모두 Phase 1 Core로 정확히 반영됨.

**부분 불일치 — O2 카카오톡 알림 (ICE 3위 → Phase 2)**:
- ICE Score 448로 3위임에도 Phase 1에서 구현되지 않고 Web Push로 대체됨.
- 이는 ICE 방법론의 한계를 보여주는 사례. PRD가 Ease 점수(8점)를 높게 설정했으나, 실제로는 카카오 비즈니스 API 심사·연동 복잡도·비용이 단기 구현을 저해함.
- **PM 관점 교훈**: ICE의 Ease 점수는 "내부 구현 난이도"만 반영하고 "외부 의존성(파트너 API 정책)"을 과소평가할 수 있음. 카카오 알림톡 ICE Ease는 8점이 아닌 4~5점으로 재평가가 적절했을 것.

**암묵적 우선순위 변경 — Web Push 추가**:
- PRD OST에 "Web Push (VAPID)"는 별도 기회영역이나 Solution으로 명시되지 않았으나, 구현 단계에서 O2(카카오)의 현실적 대안으로 채택됨.
- 이는 Phase 1 범위에서 올바른 의사결정이나, PRD OST 트리에 반영되지 않아 Discovery 문서와 구현 간 갭 발생.

**구현 모듈 순서 (실제 개발 시퀀스)**:
```
M1: 기반 스택(인증/마스터) → M2: 티켓 State Machine → M3: 배치+워크플로우+알림 → M4: 대시보드+온보딩
```
이 순서는 ICE보다 기술적 의존 관계(하위 계층 → 상위 계층)를 반영한 합리적 구현 순서.

## 5. 신규 기회영역 발견

구현 과정에서 PRD OST에 명시되지 않았으나 실제 코드와 설계 결정 과정에서 새롭게 드러난 기회영역을 기술한다.

---

### NO1 — 배치 시스템 신뢰성이 경쟁 차별화 요소 (신규 발견)

**발견 경위**: PRD OST에 배치 자동화는 Solution 수준에서 언급됐으나, 구현 결과 BullMQ 10개 배치 잡(DLQ, Exponential Backoff, Graceful Shutdown)이 글로벌 경쟁사 대비 실질적 차별점임이 확인됨.

**기회**: "자동접수 → 지연감지 → 에스컬레이션 → 만족도 자동종료" 4단계 자동화 루프는 지원팀 야간/주말 부재 시에도 SLA를 보호함. 이는 Zendesk Free/Pro에서 제공하지 않는 "한국형 근무시간 기반 자동화"임.

**잠재 OST 위치**:
```
[Opportunity] 야간/주말 서비스 공백 자동 보호
  └── [Solution] 근무시간 기반 배치 자동화 (구현 완료)
  └── [Solution] 공휴일 캘린더 연동 SLA 계산 (구현 완료)
```

---

### NO2 — 온보딩 완료율이 PLG 핵심 지표로 부상 (신규 발견)

**발견 경위**: 온보딩 마법사(F1.11) 구현 시 `OnboardingState` 모델로 진행 상태를 DB에 저장하는 설계가 채택됨. 이는 단순 UX 기능이 아닌 PLG(Product-Led Growth) 활성화 측정 도구임.

**기회**: `OnboardingState.completedSteps` 데이터를 활용하면 "Step 2(회사설정)에서 이탈률이 높다"는 인사이트 도출 → 개선 실험 설계 가능. PRD Success Criteria SC-13(온보딩 완료율 70%)이 실제 측정 가능한 지표가 됨.

**잠재 OST 위치**:
```
[Opportunity] 초기 설정 복잡도로 인한 활성화 실패
  └── [Solution] 온보딩 상태 저장 + 이탈 지점 분석 (구현 완료)
  └── [Solution] 이탈 지점 기반 인앱 가이드 추가 (미구현)
```

---

### NO3 — 모바일 승인 UX가 Persona 3(총무팀) 확보 핵심 (신규 발견)

**발견 경위**: PRD Persona 3(박정호, 총무, 낮은 Tech Savviness, 모바일 중심)에 대응하는 모바일 UX 요구사항이 구현됨 — 하단 고정 액션바, 사이드바→하단 탭 전환, 44×44px 터치 타겟. 그러나 PRD OST에는 이 Persona를 직접 다루는 기회영역이 없음.

**기회**: 총무/시설관리 부서는 현재 Beachhead가 아니지만, 모바일 UX가 준비된 상태에서 "모바일 원터치 승인"을 마케팅하면 IT 스타트업 외 제조/금융 부서 확장 가속 가능.

**잠재 OST 위치**:
```
[Opportunity] 비IT 부서의 모바일 기반 서비스 요청 관리
  └── [Solution] 모바일 하단 탭 + 고정 액션바 (구현 완료)
  └── [Solution] 모바일 전용 승인/반려 원터치 뷰 (부분 구현)
```

---

### NO4 — DLQ 관리 인터페이스가 SLA 보증의 신뢰 증거 (신규 발견)

**발견 경위**: `GET /api/admin/jobs` + `POST /api/admin/jobs/{jobId}/retry` DLQ 관리 API 구현. 배치 실패 시 관리자에게 `BATCH_JOB_FAILED` 알림 발송. 이 기능이 단순 운영 도구가 아닌 "서비스 신뢰도 가시화 도구"임이 구현 후 확인됨.

**기회**: Enterprise 세그먼트(금융/공공)에서 "배치 실패 추적 및 복구 로그"는 감사 요구사항. 현재 DLQ UI가 관리자 전용이지만, 고객사에게도 SLA 이행 증거로 제공 가능.

---

### NO5 — Web Push 21종 알림이 "알림 피로" 관리 기회 노출 (신규 발견)

**발견 경위**: 21개 NotificationType이 구현되면서 사용자가 원하지 않는 알림을 수신할 가능성이 확인됨. 현재 구현에 알림 종류별 수신 설정 기능이 없음.

**기회**:
```
[Opportunity] 과도한 알림으로 인한 Push 구독 해지
  └── [Solution] 알림 타입별 수신 설정 (미구현 — Phase 2 후보)
  └── [Solution] 알림 묶음 발송(digest) 옵션 (미구현)
```

## 6. 종합 평가 및 권고사항

### 6.1 종합 평가

| 검증 영역 | 평가 | 점수 |
|---------|------|:----:|
| OST 실현도 (Phase 1 기준) | Phase 1 범위 내 핵심 Solution 구현 완료. ICE 상위 5개 기회영역의 필수 Solution 전체 반영 | 78% |
| 가정 검증 준비도 | A1/A4/A6 즉시 검증 가능 상태. A2/A5는 Phase 2~3 의존 | 57% |
| ICE vs 구현 정합성 | 7/8 기회영역 ICE 순위와 구현 Phase 일치. O2 카카오 1개 부분 불일치 | 88% |
| 실험 실행 가능성 | 5개 실험 중 3개(E3/E4/E5) 즉시 실행 가능 | 60% |

**전체 Discovery 전략 실현도**: **71%** (Phase 1 범위 기준)

Phase 1 MVP 범위 내에서 Discovery 전략이 충실히 반영됨. 미실현 항목(카카오/AI/마이그레이션/결제)은 전략적 Phase 이연으로, 실패나 갭이 아닌 의도된 범위 관리의 결과임. 전체 Discovery 전략 관점에서는 Phase 2~3까지 포함하면 미실현 항목이 모두 로드맵에 포함되어 있어 완전성 높음.

### 6.2 즉시 실행 권고 (Beta 출시 전)

**P0 — 실험 실행 (2주 내)**

1. **E5 워크플로우 Hallway Test**: 잠재 고객 5~10명 대상, 시드 계정으로 연기요청/완료요청 태스크 완료율 측정. A6 검증과 동시에 UX 개선 포인트 발견 가능.

2. **E3 Van Westendorp 조사**: Google Forms로 즉시 실행 가능. Beta 신청자 모집 채널 오픈 직후 연동. A4 검증에 필수.

**P1 — 측정 인프라 추가 (Beta 출시 전)**

3. **업그레이드 프롬프트 이벤트 로깅**: 5개 업그레이드 조건별 노출/클릭 이벤트를 별도 테이블에 기록하는 Analytics 후크. A4 검증의 핵심 데이터.

4. **OnboardingState 이탈 지점 분석 쿼리**: 각 Step별 완료율 집계 쿼리 추가. NO2 기회영역의 즉각적 활용.

5. **Web Push 응답 시간 추적**: 알림 발송 후 해당 티켓 최초 접근까지 소요 시간을 Notification 모델에 `respondedAt` 필드 추가 검토. Phase 2 A/B 테스트 베이스라인.

  └── 알림 타입별 수신 설정 (Phase 2 후보)

### 6.5 Discovery 방법론 회고

**잘 작동한 것**:
- 5-Step Discovery Chain이 Phase 1 범위 결정에 명확한 근거 제공
- ICE 매트릭스가 O1(한국어)/O3(승인)/O4(데이터 주권)의 Phase 1 우선 구현을 지지
- OST가 개발팀에게 "왜 이 기능을 만드는가"의 맥락 제공 — Option C 아키텍처 선택과 일관성

**개선이 필요한 것**:
- **ICE Ease 과대평가**: 카카오톡 알림 Ease=8은 외부 API 의존성을 반영하지 못함. Ease 평가 시 "외부 파트너 의존성" 별도 리스크 항목 필요
- **실험 선행 부재**: PRD가 "Beta 출시 전 A1, A4 최소 1개 실험 완료"를 권고했으나 미착수. 구현 완료 후 실험 실행이라는 순서는 Discovery 원칙(학습 우선) 역행
- **Web Push 대체 결정의 OST 미반영**: 카카오→Web Push 대체는 올바른 결정이나 OST 트리에 기록되지 않아 미래 팀원이 결정 근거를 추적하기 어려움
